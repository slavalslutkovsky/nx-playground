# Databases

Source-of-truth schemas, migrations, and cluster artifacts for every Postgres database in this monorepo. One folder per database.

## Layout

```
manifests/db/
├── README.md                                  # this file
├── db.just                                     # recipes (imported by /justfile)
└── <db>/                                       # one folder per database
    ├── schema.sql                              # canonical desired state
    ├── seed.sql                                # local-dev only (optional)
    ├── migrations/                             # versioned, sqlx-compatible
    │   ├── <ts>_<name>.up.sql
    │   └── <ts>_<name>.down.sql
    └── k8s/                                    # cluster artifacts (Atlas Operator)
        ├── migrations-configmap.yaml          # generated from migrations/ (committed)
        ├── atlas-migration.yaml                # AtlasMigration CR (hand-written)
        └── kustomization.yaml                  # bundles configmap + CR
```

Currently registered DBs: `zerg`. Add `protos` (or any other DB) by creating a sibling folder with the same shape.

## Conventions

- **Folder name == Postgres database name.** `zerg/` → `zerg` database. No translation layer.
- **`schema.sql` is the source of truth.** It describes the *desired* state of the database. Migrations are the *path* to that state.
- **Migrations are reversible.** Every change writes both `<ts>_<name>.up.sql` and `<ts>_<name>.down.sql`. Atlas only applies `.up.sql`; sqlx supports both for local rollback.
- **Postgres extensions/functions live inside each `schema.sql`.** No shared `common/` layer — extensions are per-database in Postgres anyway, and three lines of repetition beats an include mechanism.
- **Generated artifacts are committed.** `k8s/migrations-configmap.yaml` is regenerated by `pg-cli` and lives in git so Flux can apply it without runtime tooling.
- **Naming for cluster artifacts:** ConfigMap = `<db>-migrations`, AtlasMigration CR = `<db>-migration`.

## Workflow

### Editing the schema

1. Edit `manifests/db/<db>/schema.sql` (the desired state).
2. Create a migration that takes the live DB from current state to the new state:
   ```
   just migrate-add <db> <name>
   ```
   This creates `<ts>_<name>.up.sql` and `<ts>_<name>.down.sql` under `manifests/db/<db>/migrations/`.
3. Write the SQL in both files (`up` applies the change, `down` reverses it).
4. Apply locally and verify:
   ```
   just migrate <db>
   just migrate-validate <db>   # diffs schema.sql vs migrations result
   ```
5. Regenerate the cluster ConfigMap:
   ```
   just gen-migrations-configmap <db>
   ```
6. Commit `schema.sql`, the new migration files, and `k8s/migrations-configmap.yaml`.

### Adding a new database

```
mkdir -p manifests/db/<db>/{migrations,k8s}
$EDITOR manifests/db/<db>/schema.sql           # write canonical schema
just migrate-add <db> initial                   # creates timestamped up/down stubs
$EDITOR manifests/db/<db>/migrations/<ts>_initial.up.sql
$EDITOR manifests/db/<db>/migrations/<ts>_initial.down.sql
just gen-migrations-configmap <db>              # generates k8s/migrations-configmap.yaml
```

Then:
- Append `<db>-local` and `<db>-cluster` lines to the `_db-url` case in `db.just`.
- Hand-write `manifests/db/<db>/k8s/atlas-migration.yaml` (copy from `zerg/`, swap names + secret reference).
- Hand-write `manifests/db/<db>/k8s/kustomization.yaml` (copy from `zerg/`).
- Add a Flux Kustomization in the gitops-v1 repo pointing at `./manifests/db/<db>/k8s` (copy `zerg-migrations-dev`).

No edits required outside the new folder.

## Recipes

All recipes live in `db.just` (imported by the root `/justfile`). Per-DB recipes take `<db>` as their first argument.

### Discovery
| Recipe | What it does |
|---|---|
| `just db-list` | List all configured databases |

### Local development (docker-compose Postgres on `:5432`)
| Recipe | What it does |
|---|---|
| `just db-fresh <db>` | Drop + recreate `<db>`, apply `schema.sql`, apply `seed.sql` |
| `just db-seed <db>` | Re-apply `seed.sql` only |
| `just db-inspect <db>` | `pg_dump` the live schema for comparison |
| `just reset-db` | Nuke docker volumes + reapply every DB |

### Migrations
| Recipe | What it does |
|---|---|
| `just migrate-add <db> <name>` | Create `<ts>_<name>.up.sql` and `<ts>_<name>.down.sql` |
| `just migrate <db>` | Apply pending migrations to local DB |
| `just migrate-revert <db>` | Revert the last migration locally |
| `just migrate-cluster <db>` | Apply migrations to kind cluster (port-forward `:5433`) |
| `just migrate-status <db>` | Show which migrations are applied |
| `just migrate-baseline <db>` | Mark all migrations as applied (use after `db-fresh`) |
| `just migrate-test <db>` | Drop → create → migrate → seed (validates the full prod path) |
| `just migrate-validate <db>` | Diff `schema.sql` vs the result of running migrations |

### Cluster ConfigMap generation (Atlas Operator)
| Recipe | What it does |
|---|---|
| `just gen-migrations-configmap <db>` | Generate `<db>/k8s/migrations-configmap.yaml` (versioned mode) |
| `just gen-schema-configmap <db>` | Generate `<db>/k8s/schema-configmap.yaml` (declarative mode) |
| `just gen-all-configmaps` | Loop over every DB folder, regenerate |

### Cluster status
| Recipe | What it does |
|---|---|
| `just cnpg-status [ns]` | Show CNPG cluster + AtlasMigration status |
| `just cnpg-logs [ns]` | Tail CNPG logs |

## Cluster delivery

Production migrations are applied by the [Atlas Kubernetes Operator](https://atlasgo.io/integrations/kubernetes/operator) via an `AtlasMigration` custom resource per DB.

Per-DB Flux delivery (one Kustomization per DB in the gitops-v1 repo) reconciles `manifests/db/<db>/k8s/` to the cluster. The operator:
1. Connects to the database via the secret referenced in `urlFrom.secretKeyRef`.
2. Reads the migrations ConfigMap (committed, generated by `pg-cli`).
3. Applies pending migrations in version order, tracked in the `atlas_schema_revisions` table.
4. Reports Ready when caught up — only then does the dependent app Kustomization start.

Use `baseline: "<version>"` on the `AtlasMigration` spec when adopting an existing populated DB so Atlas skips already-applied migrations instead of failing on duplicate `CREATE TABLE`.

## Tooling

- **`pg-cli`** — generates ConfigMaps from `schema.sql` / `migrations/`. Source lives in `~/dotconfig/crates/pg-cli`. Install via `cargo install --path ~/dotconfig/crates/pg-cli` (or `just install-bins` from `~/dotconfig`).
- **`sqlx`** — applies migrations locally. Install via `cargo install sqlx-cli`.
- **`just`** — task runner. Install via `cargo install just` or `brew install just`.
- **Postgres 18** — local docker-compose and the production CNPG cluster both run PG18. `uuidv7()` is a built-in (no extension required).
